# Архитектура Mushketon Live Scoring

## Схема потока данных

```
Судья открывает секретную ссылку (?token=...)
        │
        ▼
judge/index.html (GitHub Pages)
        │  читает ключи из ../config.js
        │  INSERT / UPDATE с заголовком x-judge-token
        ▼
Supabase DB (PostgreSQL)
        │  Realtime broadcast (WebSocket push, без polling)
        ▼
scoreboard/index.html (GitHub Pages)
        │  читает ключи из ../config.js
        ▼
Зрители
```

## Сервисы

| Сервис | Назначение | Тариф |
|--------|-----------|-------|
| GitHub | хранение кода | бесплатно |
| GitHub Pages | хостинг HTML | бесплатно |
| Supabase | БД + Realtime + RLS | бесплатно |

## Авторизация судьи

- Токен передаётся в URL: `?token=...`
- JS читает токен и добавляет в каждый запрос заголовок `x-judge-token`
- Supabase RLS проверяет заголовок через функцию `get_judge_token()`
- Токен хранится ТОЛЬКО в: Supabase (функция) + URL у судьи
- Смена токена: `sql/rotate-token.sql` (код репозитория не меняется)
- Anon key в `config.js` — публичный по дизайну, защита через RLS

## Конфигурация

Ключи Supabase вынесены в корневой `config.js` (`window.MUSHKETON_CONFIG`).
Обе страницы подключают его через `../config.js` — ключи меняются в одном
месте. Параметры серий/раундов для обоих типов соревнования зашиты в JS.

## RLS политики

```
Все (анонимы)        → только SELECT
Запрос с токеном     → SELECT + INSERT + UPDATE + DELETE
```

## Типы соревнований

Система поддерживает два несвязанных типа соревнования, хранящихся в поле
`competitions.type`:

| `type` | Формат | Строки `shooters` | Колонки ввода | Серии/раунды |
|--------|--------|-------------------|---------------|-----------------|
| `pairs` | 4 пары / 8 стрелков, группировка по щитам | 2 строки на щит | `s1..s6` | `5 5 5 3 3 3` |
| `final` | индивидуальный финал, 8 стрелков, без пар | 1 строка на стрелка | `s1..s9` | `5 5 2 2 2 2 2 2 2` |

Геттеры `seriesLabels()`/`seriesCount()` (judge) и `seriesCountFor()` (scoreboard)
выбирают правильный набор каждый раз по `competition.type` — при создании
нового соревнования другого типа (без перезагрузки страницы) таблица ввода
перестраивается корректно.

**Выбывание в финале не реализовано в коде.** Рейтинг всегда просто сортируется
по сумме введённых очков; судья сам определяет выбывание по рейтингу и
просто не вводит дальнейшие раунды выбывшему — подробнее правило отсечки
смотри в [docs/usage.md](usage.md#индивидуальный-финал-8-стрелков).

## База данных

### Таблица `competitions`

| Поле | тип | Описание |
|------|-----|----------|
| `id` | uuid | PK |
| `date` | date | дата (проставляется автоматически) |
| `is_active` | boolean | активное соревнование |
| `type` | text | `'pairs'` или `'final'` (по умолчанию `'pairs'`) |

### Таблица `shooters`

| Поле | тип | Описание |
|------|-----|----------|
| `id` | uuid | PK |
| `competition_id` | uuid | FK → competitions |
| `target` | integer | номер щита (`pairs`) или порядковый номер (`final`), 1–8 |
| `name` | text | фамилия |
| `position` | integer | всегда 1 (один стрелок на щит) |
| `s1`..`s6` | decimal | суммы серий (тип `pairs`) |
| `s7`..`s9` | decimal | суммы дополнительных раундов (тип `final`, всегда `null` для `pairs`) |

## Логика пар (тип `pairs`)

Фиксированный формат: **4 пары / 8 стрелков** (щиты 1–8). Пара = два соседних щита:

```
Пара 1 = щит 1 + щит 2
Пара 2 = щит 3 + щит 4
Пара 3 = щит 5 + щит 6
Пара 4 = щит 7 + щит 8
```

Итого пары = сумма всех `s1..s6` обоих стрелков.
Группировка в JS: `Math.floor((target - 1) / 2)`

## Индивидуальный финал (тип `final`)

8 стрелков, без группировки — 1 строка таблицы на стрелка. Итого = сумма `s1..s9`.
Подробнее о работе судьи и правиле отсечки — в [docs/usage.md](usage.md#индивидуальный-финал-8-стрелков).

## Realtime

Scoreboard не опрашивает базу по таймеру. Supabase пушит изменения через WebSocket:

```
Судья нажал Enter
  → Supabase UPDATE shooters
    → Realtime broadcast
      → scoreboard получает событие
        → пересчитывает рейтинг
          → перерисовывает таблицу (~100–300 мс)
```
